Bash Shell - Terminal - Command Line

In laymans terms, a shell means any program that is used to send commands to the operating system. A shell provides a means to interact with the kernel, and the 'lower levels' of the operating system.

If a software engineer wants to write a programme that requires the ability to interact with the underlying operating system (that would be all of them), for example to read/write from a file, or to play sound through the speakers, that is where a shell comes in handy. The author does not need to write the code from scratch, in fact in pretty much all OS's this is prohibitted, as these functions need to be standardised. The limited functions of the underlying operating system are all accessible through commands, which can be sent from a shell.

The Bash Shell is a Unix shell and command language written by Brian Fox. The terminal is the programme that provides the interface to the bash shell, allowing the user to interactively send commands directly to the OS. When we do this, we are interacting with the Command Line.

If you want to understand how a general computer application functions, you are usually going to need to understand how it is interacting with the OS, that is the fundamental reason as to why it is important to have the necessary knowledge of UNIX commands, to know what you can ask from your OS and how you can ask it.

If you are here because you want to be a penetration tester - then it is absolutely 100% necessary to be comfortable with the command line of your chosen OS.

Don't worry, the whole point of the bandit game is to get you comfortable using the command line. Right now generally you can think of the terminal programme as an extremely 'raw' and stripped back interface into the heart of your computer.

The man command and SSH - Secure Shells.

The man (short for manual) command can be used to gain information about other UNIX commands....

Since we will be playing the wargames remotely - much like in real life where the target computer is likely to be a remote system - we are going to need to have the ability to run a remote shell on the target computer (in this case the over the wire servers).

To do this, we will be utilising the power of SSH.

Try typing man ssh into your terminal, you will recieve a whole bunch of information about the SSH command. The display is in a pager format, hit b to go back, and hit q to return to the shell programme.

To summarise the... shell that is running on a remote computer, to which you maintain a connection using the secure socket layer.

The result of this is that every command we type into our terminal will be sent to the connected system. as if we were physically sitting at the machine and running a shell (a command line/terminal session)

You can learn about SSH in more depth here.

We are going to learn by doing and dive right in, lets create an SHH connection to the over the wire servers.

Finding the password for bandit2

Again, we'll start by running ls to exanmine the contents of the directory, we see that the directory contains a file called: -

In UNIX, '-' is reserved as a special character. Specifically the '-' symbol is used to prefix an option following a UNIX command. (It's also used to switch between input and output, as well as filling the traditional role of arithmetic subtraction.)

Remember back to when we sent out our ssh command: ssh bandit0@bandit.labs.overthewire.org -p 2220. Part of our command contains the component '-p 2220', here we were supplying 2220 as the value of the port option for the ssh command. Since '-' is interpretted differently than other characters, This leaves us with a problem. In the previous example we used 'cat (filename)' in order to print the contents of the file. if we try to input '-' as the our filename, the command line interprets this as us telling it we will supply an option to the cat command, which we never do.

If you have a background in coding, you may already be familiar with the concept of special characters. Remember that bash is not just a 'brand' of shell, but also a language in itself, logically it has a whole bunch of special characters that are reserved to have specific role. You can learn more about special characters in UNIX here.

Luckily the fact that people would inevitably use the special characters that are set aside in the bash language to name their own crap didn't make it past Brian and the other developers, and the language provides a simple work around.

Providing ./ lets the command you are providing the exact location of an executable? file.
