Welcome To The Over-The-Wire Tutorials

Introduction
‘Over The Wire’ is my favourite brand of online capture-the-flag style wargames, the flags arerepresented by a password stored somewhere on the current level that can be used to log into the next level.

Currently over the wire offers 12 online games, and 4 downloadable games, more than enough to keep a you entertained for a while.

The games cover a range topics, from simple things like navigating directories and files on a remote server, to exploiting common web vulnerabilities, Buffer Overflows and much much more.

If you are purely looking for solutions or to get to the end of the games as quickly as possible, then this probably isn't the place for you. The purpose of these walkthroughs is to discuss the underlying methods behind each solution, our main focus is learning.
The General Goal
Each level presents you with the same abstract task: You are given access to a remote shell within a user account on the over the wire servers. Essentially we have access to a limited/restricted part of the network, and our goal is to obtain information or use knowledge of common vulnerabilities in order to gain access to the next part of the network.

Helpfully, the levels run logically in order and you generally have some idea of what type of vulnerability you are looking for.

Bash Shell - Terminal - Command Line

What Is A Shell?
In laymans terms, a shell means any program that is used to send commands to the operating system. A shell provides a means to interact with the kernel, and the 'lower levels' of the operating system.

The Bash Shell is a Unix shell and command language written by Brian Fox. The terminal is the programme that provides the interface to the bash shell, allowing the user to interactively send commands directly to the OS. When we do this, we are interacting with the Command Line.

Are They Useful?
If a software engineer wants to write a programme that requires the ability to interact with the underlying operating system (that would be all of them), for example to read/write from a file, or to play sound through the speakers, that is where a shell comes in handy.

The author does not need to write the code from scratch, in fact in pretty much all OS's this is prohibitted, as these functions need to be standardised. The limited functions of the underlying operating system are all accessible through commands, which can be sent from a shell.
Do I Need To Know This Shit?
If you want to understand how a general computer application functions, you are usually going to need to understand how it is interacting with the OS, that is the fundamental reason as to why it is important to have the necessary knowledge of UNIX commands, to know what you can ask from your OS and how you can ask it.

If you are here because you want to be a penetration tester - then it is absolutely 100% necessary to be comfortable with the command line of your chosen OS.

Don't worry, the whole point of the bandit game is to get you comfortable using the command line. Right now generally you can think of the terminal programme as an extremely 'raw' and stripped back interface into the heart of your computer.



Your OS will have a native programme to initiate a shell, If you're on a windows system you want the Command Prompt programme, and on MacOS it's the application called Terminal.

Hopefully you have managed to open up your shell programme. From here, we can send commands, and also recieve output to the screen.



The man command and SSH - Secure Shells.

The first command we will utilise is the man command (short for manual). It can be used to gain information about other UNIX commands. The command man [command-identifier] essentially asks the terminal to tell you all about the command you specified.

Since we will be playing the wargames remotely - much like in real life where the target computer is likely to be a remote system - we are going to need to have the ability to run a remote shell on the target computer (in this case the over the wire servers).

To do this, we will be utilising the power of SSH.

Try typing man ssh into your terminal, you will recieve a whole bunch of information about the SSH comand. The display is in a pager format, hit q to return to the shell programme.

To summarise, initiating an SSH connection provides you with a shell that is running on a remote computer, to which you maintain a connection using the secure socket layer.

The result of this is that every command we type into our terminal will be sent to the connected system. as if we were physically sitting at the machine and running a shell (a command line/terminal session)

You can learn about SSH in more depth here.

We are going to learn by doing and dive right in, lets create an SHH connection to the over the wire servers.

Finding the password for bandit2

Again, we'll start by running ls to exanmine the contents of the directory, we see that the directory contains a file called: -

In UNIX, '-' is reserved as a special character. Specifically the '-' symbol is used to prefix an option following a UNIX command. (It's also used to switch between input and output, as well as filling the traditional role of arithmetic subtraction.)

Remember back to when we sent out our ssh command: ssh bandit0@bandit.labs.overthewire.org -p 2220. Part of our command contains the component '-p 2220', here we were supplying 2220 as the value of the port option for the ssh command. In the previous example we used 'cat (filename)' in order to print the contents of the file. If we try to input '-' as the our filename, the command line interprets this as us telling it we will supply an option to the cat command, which we never do.

If you have a background in coding, you may already be familiar with the concept of special characters. Remember that bash is not just a 'brand' of shell, but also a language in itself, logically it has a whole bunch of special characters that are reserved to have a specific role. You can learn more about special characters in UNIX here.

Luckily the fact that people would inevitably use the special characters that are set aside in the bash language to name their own crap didn't make it past Brian and the other developers, and the language provides a simple work around.

Providing ./ lets the command you are providing the exact location of an executable? file.

Earlier I said we would explore the options of the ls command in more detail and I'm a man of my word. We'll start by covering how the ls command handles hidden files.

In UNIX, a hidden file is any file that begins with a dot (.) By convention, the ls command does not list hidden files.

Providing the letter a as an option to the ls command will signify that you want to list the hidden directories.

The l option is used to produce output in a UNIX long represention, in simpler terms we won't just get the name of the file but also be shown the permissions among other details.

The command that we are going to run is ls -la (most terminals accept EITHER ls -la OR ls -al).

The find command is a pretty complex function, it's worth looking into in more depth when you have the time.

Here we are using some useful options you might find yourself using regularly, it is benificial to try to remember them, but many 'cheat sheets' are available such as this

find an item whose type is file, is readable, non executable and has a size of 1033 bytes. find . -type f -readable ! -executable -size 1033c

we have supplied the -type option, with a value of f (torepresent file), following that we apply the -readable flag.

! is used to mean not, so when we sebsequently supply the -executable option we are telling the command line that we don't want to include results that are executable.

Next we supply the size option with a value of 1033c (1033 bytes).

Again the OS does all the hard work for us, a single result is printed to the screen. examiningthe file we see it does in fact contain the elusive flag.

Bandit 6 -> 7

find / -user bandit7 -group bandit6 -size 33c The file containing our password is one of the many outputted files, youmay be able to see it, since its the only line that does not contain an error message, we don't have permission to read any of the other files. A quick fix is to append 2>/dev/null to the end of our command. 2 is the standard identifier for error output.

On a linux system, dev/null is considered as a black hole or a void, anything placed in there is considered lost forever.

Once the annoying error messages havebeen sent to the abyss, we can see the name of the file we were looking for much more conveniently.
